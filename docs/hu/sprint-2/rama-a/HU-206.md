# HU-206 — Callback Transbank + Activación de Plan
**Sprint:** 2 | **Rama:** A | **Alta**

## Definición
Crear `GET /api/billing/callback` que recibe el retorno de Transbank después del pago.

## Flujo
1. Transbank redirige al usuario a `/api/billing/callback?token_ws=xxx`
2. Si `token_ws` está ausente → usuario canceló → redirect `/facturacion?status=cancelled`
3. Si existe `TBK_TOKEN` en query (abort por timeout) → redirect `/facturacion?status=timeout`
4. Llamar `webpayTransaction.commit(token_ws)`
5. Verificar `response.responseCode === 0` (transacción aprobada)
6. Si aprobada:
   - Buscar `billing_transactions` por `buy_order` → obtener `user_id` y `plan_type`
   - Calcular `end_date` = `now + PLAN_DURATIONS[plan_type]` días
   - Upsert `user_billing`:
     ```
     plan_type, plan_status='active',
     tbk_order_id = buy_order,
     tbk_token = token_ws,
     start_date = now,
     end_date = end_date
     ```
   - Actualizar `billing_transactions.status = 'approved'`, guardar response en `tbk_response`
   - Redirect → `/facturacion/success`
7. Si rechazada:
   - Actualizar `billing_transactions.status = 'rejected'`
   - Redirect → `/facturacion?status=rejected`

## Criterios de aceptación
- [ ] GET con token_ws válido → commit → actualiza plan → redirect success
- [ ] GET sin token_ws → redirect con status=cancelled
- [ ] GET con TBK_TOKEN → redirect con status=timeout
- [ ] Transacción rechazada → redirect con status=rejected
- [ ] user_billing actualizado con fechas correctas
- [ ] billing_transactions registra el resultado

## Notas de seguridad
- Validar que el buy_order existe en billing_transactions antes de activar
- Verificar que la transacción no fue ya procesada (evitar doble-commit)
- Log completo del response de Transbank para auditoría
